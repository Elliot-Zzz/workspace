🔐 嵌入式产品软件加密方案

适用于 ARM Cortex-M0/M3/M4 等资源受限 MCU

⚠️ 免责声明

本方案仅供技术参考，无法保证完全防止破解。硬件安全需结合软件防护，必要时请咨询专业安全公司。

一、背景与威胁模型

嵌入式产品面临的常见攻击手段包括：

  • 电压/时钟 glitching（故障注入） — 通过短暂干扰芯片供电或时钟，使加密验证跳过执行
  • DPA / CPA（差分功率分析） — 通过分析芯片运行时的功耗曲线推测密钥
  • 芯片 decap + 固件提取 — 物理方法去除封装，用微探针直接读取 flash 内容
  • 调试接口攻击 — 通过 SWD/JTAG 接口读取内存或控制执行

二、芯片级安全特性（选型建议）

如果预算允许，优先选择带硬件安全特性的 MCU：

STM32L4+ 系列
含 Secure Boot + 主动防篡改
NXP LPC55Sxx
内置 Secure Boot、可信区域
Infineon PSoC 6
双核架构，安全隔离区
TI MSP432E4
安全启动 + 加密引擎

三、软件层面加密方案

1. 代码混淆

将关键算法拆分、跳转嵌套，使反汇编难以理解：

  • 将核心算法拆分到多个不连续地址
  • 使用函数指针间接调用，避免静态分析
  • 插入无效跳转和垃圾代码干扰分析
  • 编译器级别混淆（如使用 LLVM-Obfuscator）

2. 运行时动态解密

关键数据不在 Flash 中以明文存储：

  • 固件关键区域使用 AES-128 加密存储
  • 启动时从独立安全区域读取密钥进行解密
  • 解密后的明文在内存中尽量短时间存活
  • 使用后立即覆写清除敏感数据

3. 校验和反调试

检测异常入侵并主动破坏：

  • 检测调试器连接（检查 IDR 寄存器）
  • 关键代码段运行时计算 CRC 校验，不匹配则擦除关键数据
  • 检测异常复位（电源 glitch 常见手段）
  • 加入"蜜罐"代码，触发即自毁

4. 时间随机化

防止通过时序分析进行 glitch 攻击：

  • 关键操作加入随机延时（微秒级随机）
  • 加密操作前加入随机等待，打破时序相关性
  • 使用随机填充使功耗曲线不可预测

5. 多级验证机制

分散验证点，降低单点突破风险：

  • 第一级：启动时验证核心固件完整性
  • 第二级：关键功能执行前二次验证
  • 第三级：定期后台校验（定时器触发）
  • 各验证密钥分开存放，禁止集中

6. 密钥分片管理

密钥不存储在单一位置：

  • 将主密钥拆分为 N 份，分别存放在不同 Flash 区域
  • 运行时通过特定算法动态拼接
  • 每片密钥可附加特定校验（如特定位置求和）
  • 升级时仅更换其中一片，增加兼容复杂度

7. 轮换密钥机制（Rotating Secret）

每次固件升级更换加密方式：

  • 每次升级生成新的加密密钥
  • 旧版本密钥在新版本中失效
  • 增加回滚难度，使历史固件无法复用
  • 与服务器配合实现一对一轮换

8. 通讯层加密

产品与服务器/APP 通信安全：

  • 使用 TLS 1.2+ 或自定义轻量级加密通道
  • 每次通信使用动态会话密钥
  • 加入设备唯一标识 + 时间戳防重放
  • 关键指令额外增加签名验证

四、总结

在资源受限的 Cortex-M0/M3 芯片上，无法使用复杂的加密方案，但可以通过多层叠加的方式提高破解成本：

  • 基础层：读保护 + 加密存储 + 密钥分片
  • 进阶层：代码混淆 + 反调试 + 时间随机化
  • 高级层：动态解密 + 多级验证 + 轮换密钥

目标是让破解成本高于产品利润，使抄袭者无利可图。
