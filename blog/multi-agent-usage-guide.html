🔬 为什么单 Agent 会遇到天花板

先说一个反直觉的事实：很多 Agent 问题不是模型不够聪明，而是架构设计不对。

Anthropic 的研究（2025）发现：LLM 的指令遵循准确度每增加 1000 token 系统提示词，下降约 8%。也就是说，把什么功能都塞进一个 Agent 里，它反而会更容易出错。

单 Agent 的三大瓶颈

  • 指令稀释：工具越多、职责越杂，Agent 越容易选错工具、忽略重要指令
  • 错误归因困难：一个 Agent 做调研+分析+写作，出了问题你不知道是哪一步的锅
  • 无法并行：顺序执行，多个任务排队等待，效率低下

多 Agent 的核心价值

  • 并行执行：独立的 Agent 可以同时工作
  • 职责分离：每个 Agent 只做一件事，失败可定位
  • 可组合性：出了问题只重跑相关 Agent，其他结果保留
  • 跨模型优化：简单任务用便宜模型，复杂任务用强模型，成本降低 60%

🏗️ 四大协作模式详解

业界成熟的多 Agent 协作有四种核心模式，各有适用场景。你的团队在不同任务下应该选择不同模式。

① Supervisor 模式（主管模式）
一个主管 Agent 统筹全局，负责拆解任务、分派给专家 Agent、汇总结果。Anthropic 的 benchmark 显示，这种模式比扁平架构在复杂推理任务上提升 23%。
  • 何时用：任务复杂、步骤多、需要动态调整执行计划
  • 代表案例：微软 Copilot、AutoGen 的 Orchestrator 模式
  • 在你的团队：Elaine 作为 PM 就是天然的主管 Agent
② Pipeline 模式（流水线模式）
顺序执行，每个 Agent 的输出是下一个 Agent 的输入。没有中央协调，流程在设计时确定。最容易调试，因为每个阶段都有可检查的中间产物。
  • 何时用：线性流程、阶段边界清晰（如：需求→开发→测试→发布）
  • 代表案例：内容生产流水线（大纲→草稿→审核→发布）
  • 在你的团队：新功能从 Sheldon 开发 → Leonard 测试 → Howard 发布
③ Debate 模式（辩论模式）
多个 Agent 独立完成同一任务，再由裁判 Agent 评判最优解。Google DeepMind 的研究表明，这种模式能将事实准确率提升 18-31%，尤其在需要多步推理的任务上效果显著。
  • 何时用：高风险决策、代码正确性要求高、战略分析
  • 代表案例：代码生成中两个 Agent 独立实现，Leonard 验证哪个更优
  • 关键：裁判 Agent 必须有明确的评分维度，不能模糊地说"选最好的"
④ Swarm 模式（蜂群模式）
Agent 之间点对点通信，没有固定层级，自主协调。最灵活，也最难控制。需要共享状态机制（workspace 文件、消息队列），否则会陷入混乱。
  • 何时用：探索性研究、多路径并行验证、创意头脑风暴
  • 代表案例：OpenAI Swarm 框架（但也因为控制难度大而争议不断）
  • 在你的团队：Raj（文档）+ Howard（运维）联合调研新技术选型时可以用

⚠️ 常见坑与避坑指南

坑 1：Supervisor 变成"超级 Agent"

最常见的反模式：把所有职责都塞给主管 Agent，它又要分析需求、又要写代码、又要审核——变成一个臃肿的单 Agent。

✅ 正确做法

主管 Agent 只做协调和汇总，真正的执行交给各专家 Agent。Elaine 的定位是"指挥"，不是"执行所有事"。

坑 2：Agent 之间传信息"靠猜"

Agent A 输出了一堆内容，Agent B 需要从中提取自己需要的信息——但这种"默契"很容易出错。

✅ 正确做法

定义清晰的交接契约。比如 Sheldon 输出的技术方案，必须包含固定的几个字段：方案概述、风险点、依赖项、测试计划。Leonard 读取这些固定字段，而不是自己从全文里找。

坑 3：过度并行反而更慢

A16z 的研究指出：最优 Agent 数量是 3-7 个。超过 7 个，协作开销会超过分工带来的收益。

你的 5 个 Agent 正好在合理范围内，但要注意：不是所有任务都需要 5 个 Agent 同时参与。简单任务只启动 2-3 个 Agent 就够了。

坑 4：没有共享上下文导致各说各话

每个 Agent 独立工作，没有共享的上下文，结果是 Raj 查的文档 Sheldon 根本没看到。

✅ 正确做法

在 OpenClaw 中利用共享的 workspace 文件作为"黑板"。所有 Agent 的中间产物写到同一个文件，后续 Agent 可以读取。不需要实时消息传递，用文件+通知机制就够了。

坑 5：失败没有降级机制

某个 Agent 挂了，整条流水线卡死。

💡 建议

每个 Agent 的任务应该有明确的超时和重试策略。当某个 Agent 超时或报错，跳过它继续推进，最后由 Elaine 汇总时标注"某步骤异常，需要人工介入"。

第三步：完整的 Pipeline

当你对 Agent 协作建立信心后，跑完整流程：

需求 → Raj 调研 → Sheldon 开发 → Leonard 测试 → Howard 发布 → Elaine 汇总

每次跑完记录哪里出了问题，逐步优化交接契约。

💡 核心理念

多 Agent 的价值不在于"每个 Agent 都很聪明"，而在于合理的分工边界 + 清晰的交接契约 + 有效的错误处理。你的团队设计本身没有问题，关键是让它们真正开始协作。
