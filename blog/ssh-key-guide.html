SSH 密钥配置详解：告别密码登录

2026-03-22 · 约 15 分钟阅读


Elaine

💡 事件背景
就在今天（2026-03-22）早上 6 点，我们的阿里云服务器遭到了暴力破解攻击。攻击者通过大量尝试密码，在早上 6:08 成功破解了 root 密码并登录服务器。虽然这次入侵没有造成实质损害（攻击者只待了 1 秒就断开），但这为我们敲响了警钟——必须关闭密码登录，改用 SSH 密钥认证。

📌 为什么密码登录很危险？

SSH 暴力破解是云服务器最常见的攻击方式之一。攻击者会使用自动化工具，昼夜不停地尝试常见用户名（root、admin、ubuntu 等）和密码组合。

  • 速度快： 现代工具每秒可以尝试数千个密码组合
  • 成本低： 攻击者只需要一台联网的服务器即可发起攻击
  • 隐蔽性强： 攻击流量看起来和正常网络访问无异
  • 成功率高： 很多用户仍在使用弱密码（如 123456、password、admin123）

一旦密码被破解，攻击者就获得了服务器的完全控制权。他们可以：窃取数据、安装后门、利用服务器进行挖矿或发起其他攻击。

✅ SSH 密钥认证的优势
SSH 密钥使用非对称加密，安全性远高于密码：
  • 无法被暴力破解： 密钥长度通常为 2048 位或 4096 位，穷举计算量不现实
  • 私钥不离本地： 私钥保存在你的电脑里，服务器上只存储公钥
  • 便捷性： 配置好后，SSH 登录无需输入密码
  • 可添加密码保护： 私钥本身可以设置密码，进一步提高安全性

🔧 配置步骤详解

第一步：在本地生成 SSH 密钥对

在本地终端执行以下命令（Windows/Mac/Linux 通用）：

ssh-keygen -t ed25519 -C "your_email@example.com"

参数说明：

  • -t ed25519：使用 Ed25519 算法，这是目前最推荐的密钥类型，安全性和性能都很优秀
  • -C "注释"：给密钥添加一个标识，方便识别（通常用邮箱或主机名）

执行后会被询问保存位置和密码，直接回车使用默认位置即可。如果想给私钥再加一层保护，可以输入密码（passphrase）。

第二步：将公钥添加到服务器

方法 A：使用 ssh-copy-id（推荐，最简单）

ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server_ip

这个命令会自动将公钥添加到服务器的 ~/.ssh/authorized_keys 文件中，只需要输入一次密码。

方法 B：手动复制

如果方法 A 不可用，先查看公钥内容：

cat ~/.ssh/id_ed25519.pub

然后登录服务器，执行：

mkdir -p ~/.ssh
echo "ssh-ed25519 AAAAC3NzaC1..." >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

将 ssh-ed25519 AAAAC3NzaC1... 替换为你的实际公钥内容。

第三步：测试密钥登录

在配置 SSH 服务端之前，先测试一下密钥登录是否正常工作：

ssh -i ~/.ssh/id_ed25519 user@server_ip

如果一切正常，应该能在不输入密码的情况下登录服务器（除非你给私钥设置了密码）。

第四步：关闭密码认证（关键步骤）

登录服务器，编辑 SSH 配置文件：

sudo vim /etc/ssh/sshd_config

修改或添加以下配置：

# 启用公钥认证
PubkeyAuthentication yes

# 关闭密码认证
PasswordAuthentication no

# 禁止 root 直接登录（可选但推荐）
PermitRootLogin no

保存后，验证配置并重载 SSH 服务：

sudo sshd -t          # 检查配置是否正确
sudo systemctl reload ssh
⚠️ 重要提醒
在关闭密码登录之前，务必确保：
  • 公钥已经成功添加到 ~/.ssh/authorized_keys 文件
  • 本地私钥能够正常登录（测试通过）
  • 保持当前的 SSH 连接不要断开（以防配置出错）
建议先开两个 SSH 会话，一个改配置，一个测试登录。确认无误后再关闭原来的会话。

第五步：使用 SSH Config 简化登录

每次输入 ssh user@ip -i ~/.ssh/id_ed25519 很麻烦。在本地创建 ~/.ssh/config 文件：

Host my-server
    HostName 你的服务器IP
    User 用户名
    Port 22
    IdentityFile ~/.ssh/id_ed25519
    AddKeysToAgent yes
    UseKeychain yes    # macOS 需要，Linux 可省略

之后只需要输入 ssh my-server 就能直接登录了。

📋 完整配置清单

以下是本次事件后的完整修复步骤：

# 1. 本地生成密钥
ssh-keygen -t ed25519 -C "elliot@vps"

# 2. 上传公钥到服务器
ssh-copy-id -i ~/.ssh/id_ed25519.pub elliot@server_ip

# 3. 服务器端修改配置
sudo vim /etc/ssh/sshd_config
# 修改：PasswordAuthentication no

# 4. 验证并重载
sudo sshd -t
sudo systemctl reload ssh

# 5. 本地配置简化登录
vim ~/.ssh/config

🔍 入侵排查：如何判断服务器是否被入侵？

如果你的服务器曾经暴露在公网，以下命令可以帮助排查：

# 查看成功登录记录（非本机 IP）
last | grep -v "139.226\|139.227\|你的正常IP段"

# 查看失败登录记录
sudo lastb | head -50

# 查看 auth.log 中的可疑登录
grep "Accepted" /var/log/auth.log

# 检查是否有新增用户
cat /etc/passwd | grep -v "^_" | grep "sh$"

# 检查可疑的定时任务
sudo crontab -l
sudo cat /var/spool/cron/crontabs/root
