📋 Part 1: 任务流（Task Flow）详解 — 新版本核心功能

🏭什么是 Task Flow？

Task Flow 是 OpenClaw v2026.4.2 重新引入的任务编排层，位于后台任务（Background Tasks）之上。它管理持久化的多步骤工作流，拥有自己的状态、版本追踪和同步语义。

简单理解：Task Flow = 多个后台任务的总指挥。它能让你创建一个长期运行的工作流，跨越多次 agent 调用甚至多次重启，而每个具体步骤由后台任务执行。

⚙️Task Flow 的两种同步模式

模式说明适用场景
Managed（托管模式）Task Flow 全权管理生命周期，自己创建和驱动任务自动化流水线、收件箱分类、PR审核流程
Mirrored（镜像模式）Task Flow 只观察外部创建的任务，保持同步但不自建对接已有 CI/CD、外部系统监控

🖥️CLI 命令

# 查看所有活跃和最近的任务流
openclaw tasks flow list

# 查看某个任务流的详细信息
openclaw tasks flow show 

# 取消一个正在运行的任务流
openclaw tasks flow cancel <lookup>

📊任务看板 — /tasks 命令

在任何聊天会话中输入 /tasks 即可查看该会话关联的后台任务：

/tasks

任务看板显示：

  • 活跃和最近完成的任务
  • 运行时类型（ACP / Subagent / Cron / CLI）
  • 状态（运行中 / 成功 / 失败 / 超时 / 取消）
  • 时间和进度详情
💡 状态说明
queued已创建，等待 agent 启动
runningagent 正在执行
succeeded成功完成
failed执行出错
timed_out超过超时限制
cancelled被操作员取消
lost会话消失（超过5分钟无响应）

🔔任务通知策略

每个任务可以配置不同的通知级别：

策略行为
done_only（默认）仅在任务结束时通知（成功/失败/超时等）
state_changes每次状态变化都通知
silent完全不通知
# 修改某个任务的 notification 策略
openclaw tasks notify <lookup> state_changes

🔍ClawFlow — Task Flow 的底层实现

ClawFlow 是 Task Flow 的运行时底层。对于需要跨越多个 detached 任务、但仍想保持单一所有者上下文的复杂工作流，ClawFlow 提供：

  • Flow 身份（flowId）
  • 所有者会话和返回上下文
  • 等待状态 + 小型持久化输出袋
  • 完成 / 失败 / 取消 / 阻塞 状态管理
💡 典型工作流
1. createFlow(...)         → 创建 flow
2. runTaskInFlow(...)     → 在 flow 中运行任务
3. setFlowWaiting(...)    → 等待外部事件
4. resumeFlow(...)       → 恢复 flow
5. setFlowOutput(...)    → 设置输出
6. finishFlow(...)       → 完成 flow

🏗️如何用 Task Flow 编排复杂任务？

以「每周报告」为例，一个完整的多步骤 flow：

Flow: WeeklyReport
  Step 1: 获取数据  →  Task (cron trigger)
  Step 2: 分析数据  →  Task (subagent, 等待 Step 1)
  Step 3: 生成报告  →  Task (等待 Step 2)
  Step 4: 发送邮件  →  Task (等待 Step 3)
  Step 5: 通知所有者 →  Task (flow 完成)

即使中间 gateway 重启，flow 的状态被持久化，重启后自动从断点恢复。

⚡">3. Subagent 并行任务（强大但你可能还没用过）

Subagent 允许你在一次对话中同时派遣多个后台 agent执行不同任务：

/subagents spawn sheldon "帮我写一个嵌入式bootloader代码"
/subagents spawn leonard "测试 sheldon 写的 bootloader 是否有 bug"

两个 subagent 并行运行，完成后各自返回结果。

关键参数：

  • --model — 为 subagent 指定模型（可以用更便宜的模型）
  • --thinking — 为 subagent 指定思考深度
  • --timeout — 设置超时（默认 900 秒）
💡 编排模式示例

主 agent 分配任务给 subagent A 和 B → A/B 并行执行 → 结果返回主 agent → 主 agent 汇总并报告给你

🔒">5. 持续订单（Standing Orders）— 实现真正的自主代理

这是最容易被忽视的高级功能。Standing Orders 让你定义永久性的操作授权，agent 在没有你干预的情况下自主执行。

区别：

  • 每次提醒 — "周五发周报" → 你每次都要说
  • Standing Order — "你负责周报，每周五执行" → 一次授权，自主演进

适用场景：

  • 每天自动整理收件箱
  • 监控股票异常波动并自动推送
  • 定时健康检查服务器
  • 自动备份重要文件
✅ 推荐：股票监控升级为 Standing Order

你的股票监控可以升级为 Standing Order：定义触发条件（±5%）、授权动作（推送钉钉）、升级规则（异常时立即通知）。

🪝">6. Webhook + Hooks — 外部事件驱动

OpenClaw 可以作为 HTTP webhook 端点，接收外部事件：

# 触发一个系统事件
curl -X POST http://127.0.0.1:18789/hooks/wake \
  -H 'Authorization: Bearer SECRET' \
  -H 'Content-Type: application/json' \
  -d '{"text":"新PR到达","mode":"now"}'

# 触发一个 agent 任务
curl -X POST http://127.0.0.1:18789/hooks/agent \
  -H 'Authorization: Bearer SECRET' \
  -H 'Content-Type: application/json' \
  -d '{"message":"分析这个PR","name":"PRReview"}'

可用场景：

  • GitHub PR/Issue 事件 → 自动分析
  • 服务器监控告警 → 自动诊断
  • 定时外呼 → 任何 HTTP 触发器

🛡️">10. 安全加固（你的配置需要关注）

⚠️ 你的安全审计发现了 3 个严重问题

根据 openclaw security audit 的结果，你的配置存在以下高风险：

  • CRITICAL: groupPolicy: "open" 允许任何人执行操作
  • CRITICAL: 钉钉 DM 完全开放（dmPolicy: "open"）
  • WARN: 无认证速率限制（暴力破解风险）

建议立即修复：

# 限制钉钉 DM 为配对模式
/config set channels.dingtalk.accounts.main.dmPolicy "pairing"

# 启用速率限制
/config set gateway.auth.rateLimit '{"maxAttempts":10,"windowMs":60000,"lockoutMs":300000}'

# 收紧群组策略
/config set channels.dingtalk.accounts.main.groupPolicy "allowlist"

🔐 Part 5: 你的安全配置修复方案

⚠️ 立即需要修复的问题

运行以下命令（每次一条）：

# 1. 限制钉钉 DM 为配对模式（需要先让用户配对）
/config set channels.dingtalk.accounts.main.dmPolicy "pairing"

/* 
   注意: 改为 pairing 后，未配对的联系人将无法发消息。
   如果你需要保持开放，先设置为 allowlist:
   /config set channels.dingtalk.accounts.main.dmPolicy "allowlist"
   /config set channels.dingtalk.allowFrom ["你的钉钉ID"]
*/

/* 2. 启用网关速率限制 */
/config set gateway.auth.rateLimit='{"maxAttempts":10,"windowMs":60000,"lockoutMs":300000}'
💡 安全配置说明

将 groupPolicy 从 "open" 改为 "allowlist" 可以确保只有授权用户才能在群里@机器人。钉钉的 dmPolicy 建议使用 "pairing"（需要配对审批）。
