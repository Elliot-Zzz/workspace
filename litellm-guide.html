🔍 什么是 LiteLLM？

LiteLLM 是一个开源的大模型统一调用框架，通过提供标准化的接口，让你可以用相同的方式调用 100+ 种不同的大模型服务。

支持的模型包括但不限于：

  • OpenAI：GPT-4、GPT-3.5
  • Anthropic：Claude 3.5、Claude 3
  • Google：Gemini Pro、Gemini Ultra
  • AWS：Bedrock（Claude、Llama、Titan 等）
  • Azure：Azure OpenAI
  • 开源模型：Llama 3、 Mistral、Qwen 等
  • 国内模型：MiniMax、Moonshot、DeepSeek 等
🎯 一句话理解

LiteLLM = 大模型的「万能翻译器」，不管你说的是什么语言（调用哪个模型），它都能帮你统一处理。

🤔 为什么需要统一调用框架？

直接调用的问题

如果你只用一个大模型，直接调用确实最简单。但现实场景往往更复杂：

  • ❌ 多模型切换麻烦：每个模型的 API、参数、响应格式都不一样
  • ❌ 无法统一监控：不同模型的用量、成本、延迟分散管理
  • ❌ 缺乏容错机制：某个模型挂了怎么办？手动切换？
  • ❌ 难以做负载均衡：高并发时无法分发请求
  • ❌ 代码耦合严重：换模型需要改大量业务代码

LiteLLM 解决什么问题？

  • ✅ 统一接口：一套代码调用所有模型
  • ✅ 智能路由：自动选择最优模型、负载均衡
  • ✅ 成本控制：预算管理、用量统计
  • ✅ 容错备份：模型故障时自动切换
  • ✅ 日志追踪：完整的请求日志和监控
  • ✅ 简化部署：一行命令启动代理服务

⚙️ 核心功能详解

1. 统一 API 接口

不管你调用哪个模型，LiteLLM 返回的响应格式都是统一的：

# 调用 OpenAI
litellm.completion(model="gpt-4", messages=[...])

# 调用 Claude
litellm.completion(model="claude-3-opus-20240229", messages=[...])

# 调用本地 Llama
litellm.completion(model="ollama/llama3", messages=[...])

# 调用 MiniMax
litellm.completion(model="minimax/MiniMax-M2.5", messages=[...])

# 调用 DeepSeek
litellm.completion(model="deepseek/deepseek-chat", messages=[...])

响应格式完全一致，业务代码无需修改，轻松切换模型。

2. 代理服务器模式

LiteLLM 可以作为代理服务器运行，所有模型统一入口：

# 启动代理服务
litellm

# 然后通过统一地址访问所有模型
curl http://localhost:4000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -d '{
    "model": "gpt-4",
    "messages": [{"role": "user", "content": "Hello!"}]
  }'

代理模式支持 OpenAI 兼容接口，现有应用无需修改代码即可接入。

3. 智能路由与负载均衡

LiteLLM 内置多种路由策略：

  • simple_shuffle：轮询分发请求
  • latency_based_routing：选择延迟最低的模型
  • cost_based_routing：优先使用便宜的模型
  • round_robin：循环分配

4. 多模型降级与容错

配置模型组，主模型不可用时自动切换到备用模型：

litellm:
  model_list:
    - model_name: gpt-4-turbo
      litellm_params:
        model: gpt-4-turbo
        api_key: os.environ/GPT4_API_KEY
        fallbacks: [{"model": "claude-3-opus"}, {"model": "gpt-3.5-turbo"}]

当 GPT-4 不可用时，自动切换到 Claude，然后切换到 GPT-3.5。

5. 成本控制与用量监控

  • 预算管理：设置每日/每月预算上限
  • 用量追踪：详细记录每个模型、每个用户的用量
  • 成本统计：自动计算各模型的成本
  • 超额拦截：预算用完前自动停止

🚀 快速上手

安装

pip install litellm

基础调用

import litellm

# 简单调用
response = litellm.completion(
    model="gpt-4",
    messages=[{"role": "user", "content": "Hello, world!"}]
)
print(response.choices[0].message.content)

启动代理服务

# 设置 API Key
export OPENAI_API_KEY=sk-...
export ANTHROPIC_API_KEY=sk-ant-...

# 启动服务
litellm

# 默认端口 4000，访问 http://localhost:4000

配置文件示例

# config.yaml
model_list:
  - model_name: gpt-4
    litellm_params:
      model: gpt-4
      api_key: os.environ/OPENAI_API_KEY
  
  - model_name: claude-3
    litellm_params:
      model: claude-3-opus-20240229
      api_key: os.environ/ANTHROPIC_API_KEY
  
  - model_name: deepseek
    litellm_params:
      model: deepseek-chat
      api_key: os.environ/DEEPSEEK_API_KEY
      api_base: https://api.deepseek.com

litellm_settings:
  drop_params: true
  set_verbose: True

📊 适用场景

企业级应用

  • 需要同时使用多个大模型服务商
  • 对成本控制和用量监控有要求
  • 需要高可用和故障转移
  • 需要对模型使用进行统一管理

AI 应用开发

  • 开发 AI 应用需要支持多模型切换
  • 希望降低对单一提供商的依赖
  • 需要本地部署开源模型

个人开发与学习

  • 尝试不同的模型，对比效果
  • 学习 prompt 工程
  • 本地跑开源模型

📝 总结

✅ LiteLLM 适用情况
  • 需要统一管理多个大模型
  • 对高可用和容错有要求
  • 需要精细化的成本控制
  • 希望简化多模型切换的代码
⚠️ 可能不需要 LiteLLM 的情况
  • 只使用单一模型
  • 对延迟极其敏感（每多一层都有开销）
  • 已经有成熟的自建网关

LiteLLM 是一个非常实用的工具，尤其适合需要多模型管理和企业级功能的场景。它把很多需要自己实现的复杂功能都封装好了，让你专注于业务逻辑而不是底层集成。
